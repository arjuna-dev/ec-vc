# Index Key

## Purpose

This document defines the numbering convention used by the human operator and companion agents when reading the docs folder.

It exists so docs appear in a meaningful order and so both human and AI collaborators understand why a document sits where it sits.

## Guiding Question

This document should help answer:

- Where does this document belong in the operating order?
- Is this document a root reference, architecture layer, rulebook, role, or tracker?
- Should this document be read before or after the file currently being edited?

If those questions are unclear, stop and clarify the document's role before renaming or reorganizing it.

## Folder Convention

The docs folder uses a mirrored layer/lifecycle structure.

The first folder level carries the operating layer:

- `000`
- `001`
- `010`
- `100`
- `002`
- `020`
- `200`
- `003`
- `030`
- `300`
- `999`

Human display names:

| Folder | Display Name | Meaning |
| --- | --- | --- |
| `000` | Root Authority | Root references, machine-readable structure, and archived legacy authority. |
| `001` | System Rules | Archived system-rule history and retained structure memory. |
| `010` | System Operation | Active architecture authority, product rules, game operation, and intake/ingestion operating drafts. |
| `100` | System File Guides | Guides for actual app files and future file surfaces. |
| `002` | Companion Rules | Companion base manual and companion-wide behavior rules. |
| `020` | Companion Stewards | Steward role contracts and stewardship checks. |
| `200` | Companion Role Guides | Concrete companion role guides, when needed. |
| `003` | Game Rules | Archived game-domain root rules and scoring history. |
| `030` | Game Operation | Archived game operation history. |
| `300` | Game Guides | Archived game rulebooks and historical game guides. |
| `999` | Workstream / Other | Current tracker, parked items, and coordination notes. |

Inside each layer, documents should live under lifecycle folders:

- `Active`: current operating truth that humans and companions may rely on
- `Draft`: approved direction or working material that is not fully binding yet
- `Archive`: retained history that should not be treated as active truth unless explicitly referenced

Examples:

- `docs/001/Active/001-Files.md`
- `docs/020/Active/020_File_Steward.md`
- `docs/100/Draft/100-Access_Assignments.md`
- `docs/002/Active/002-Companion_Manual.md`
- `docs/010/Active/Game_of.md`

Root exceptions:

- `docs/000-00.md` stays at root because it explains the folder system itself
- `docs/000-canonical-structure.json` stays at root because it is machine-read by app code
- `docs/000-workbook-schema-companion.json` stays at root as legacy reference only; it is not live shell payload truth

Do not move machine-read files casually.

If a root exception changes, update the code and docs that read it in the same atomic pass.

## Docs Navigation Rule

Read document paths as two signals:

- layer tells the document's operating meaning
- lifecycle tells whether the document is binding truth, working direction, or retained history

That means:

- `Active` docs are the first place to look for current operating truth
- `Draft` docs may contain approved direction, but should not be treated as fully binding until promoted
- `Archive` docs are memory and history, not current instruction, unless an active doc explicitly points to them

When a companion or human is unsure what to trust, prefer:

1. root machine-readable structure when code/canon needs it
2. `Active` docs in the relevant layer
3. `Draft` docs only for context or approved-but-not-born direction
4. `Archive` docs only for historical explanation

If a `Draft` or `Archive` document contains something that should become operating truth, promote the relevant rule into the correct `Active` document instead of relying on the old location.

## Domain + Depth Numbering Model

Status: `Active`

The numbering model shows both:

- domain
- depth inside that domain

Active map:

| Folder | Meaning |
| --- | --- |
| `000` | Root Authority |
| `001` | System Rules |
| `010` | System Operation |
| `100` | System File Guides |
| `002` | Companion Rules |
| `020` | Companion Stewards |
| `200` | Companion Role Guides |
| `003` | Game Rules |
| `030` | Game Operation |
| `300` | Game Guides |
| `999` | Workstream / Other |

The visual logic is:

- `001` is the root system rule layer
- `010` is the system operating layer
- `100` is the concrete system file-guide layer
- `002` is the root companion rule layer
- `020` is the companion steward layer
- `200` is the concrete companion role-guide layer
- `003` is the root game rule layer
- `030` is the game operating layer
- `300` is the concrete game-guide layer

This replaces the previous global layer model.

- each number carries a domain/depth relationship
- the digit position helps show where a document comes from and how concrete it is
- steward roles stay in the Companion branch even when they govern another domain

## Numbering Convention

### `000`

Root references and primitives.

Use for documents that sit above normal architecture layers and may be needed by almost everything else.

Examples:

- canonical structure
- language glossary
- runtime reference
- owner manual

### `001`

System rules.

Use for parent system rules, file-guide rules, and core bootstrap structure.

Examples:

- Files
- field classification
- File bootstrap

### `010`

System operation.

Use for rulebooks, architecture plans, product guides, and operating drafts that guide how the `001` system rules are applied.

Examples:

- record architecture master plan
- product reference guide
- intake architecture
- ingestion reconnect

### `100`

System file guides.

Use for documents that describe actual app files, file-specific rules, and future file surfaces.

Examples:

- File Guides Index
- Files
- System Files
- Access Assignments
- Companies
- Projects

### `002`

Companion rules.

Use for companion-wide behavior rules, factuality rules, and the base companion manual.

Examples:

- Companion Manual

### `020`

Companion stewards.

Use for documents that define how a steward role thinks, checks, guides, or governs.

Examples:

- Architect Steward
- File Steward
- UX Steward
- Design Steward
- Glossary Steward
- Game Steward
- Intake Steward
- Provenance Steward
- Runtime Steward

### `200`

Companion role guides.

Use for concrete companion role guides when a role becomes specific enough to browse, audit, and govern directly.

Examples:

- future Intake Assistant guide
- future Project Guide guide
- future Points Tracker guide

### `003`

Game rules.

Use for game-domain root rules, scoring primitives, and pointing-system authority.

Examples:

- Games

### `030`

Game operation.

Use for operating systems that explain how game rules are applied in practice.

Examples:

- Game Operation

### `300`

Game guides.

Use for concrete game rulebooks and game-specific guides.

Examples:

- Game Rulebook

Note: `Game Steward` remains in `020` because it is a companion steward role. The Game branch describes the game system itself, not the companion role that governs or explains it.

### `999`

Current workstream tracking.

Use for active direction, current priorities, and temporary coordination notes that should appear last.

## Steward Gateway Questions

Each steward or companion role should begin with `3` core gateway questions by default.

A role may grow to `up to 5` gateway questions when the extra questions protect against known drift, ambiguity, or implementation risk.

If a role needs more than `5`, stop and ask whether:

- the role is carrying too much responsibility
- one question belongs in a checklist instead of the gateway
- a new steward role is needed
- the architecture boundary is unclear

Gateway questions should remain short, concrete, and useful as stop conditions.

## Working Rule

The number is not decoration.

It tells us the document's operating layer.

When a document feels hard to place, that is a signal that its purpose may need to be clarified before the file is renamed.

## Collaboration Acronyms

These acronyms help the human operator and companion agents keep conversation mode clear.

### `REF`

Use when a message is answering or referring back to previous context.

Meaning:

- connect this message to the prior thread
- do not treat it as a brand-new instruction unless the user says so

### `QI`

Use for quick iteration.

Meaning:

- keep scope tight
- move quickly
- avoid over-architecting
- do not commit unless explicitly approved

### `NCT`

Use for non-commit testing.

Meaning:

- prototype or test the idea if needed
- validate the result
- pause before committing

### `STOP`

Use for a hard stop.

Meaning:

- stop the current path immediately
- do not keep iterating
- focus only on stopping, reverting, or clarifying the current issue

### `DOC`

Use when the decision should be documented but not implemented yet.

Meaning:

- write the decision down
- do not change runtime behavior unless separately requested
